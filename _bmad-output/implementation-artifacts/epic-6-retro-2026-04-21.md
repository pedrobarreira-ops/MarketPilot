## Key Insights

1. **Retro compounding is now a three-epic-long durable pattern, not a hopeful trend.** Epic 3→4, 4→5, 5→6 all shipped 100% of critical-path items in the pre-epic prep PR. Every Epic 5 commitment fired at least once in Epic 6, many fired multiple times. The loop — retro surfaces gap → gap becomes gate → gate catches the next real issue — is proven. The failure mode to watch for is gate-effectiveness decay: P3 (BAD Step 6 tweak) and T3 (CTA_URL scan) both shipped but only partially solved their targets. Next retros should track "shipped + effective" and "shipped + missed its target" separately, not just "shipped".

2. **Parallel BAD execution has a working ceiling.** 2-way (Epic 5) → 3-way (Epic 6) without regressions. 6.4 and 6.6 ran serially, correctly — 6.4 needed 6.5's error-card work to merge first (to test "header + CTA still visible in error state" consistently), and 6.6 needed all ARIA-producing stories done. BAD's dependency graph correctly identified the parallelism opportunity and the serial constraint. Trust the graph; don't force parallelism when dependencies are ambiguous.

3. **The shared invariants file has become the mandatory passthrough between stories — and that's the point.** 13 invariants, every new report.js edit had to clear them. Story 6.1's `textContent`+`createElement` no-data pattern was a direct consequence of the `innerHTML` user-interpolation rule. Story 6.2 imported the same pattern. Story 6.5 inherited it when writing `showExpiryCard`/`showFetchErrorCard`. The file is no longer a test file — it's an architectural contract enforced by tests. **Graduate pattern: project-wide invariants (Tailwind JIT, no cross-seller JOINs, api_key boundary) should all live in shared invariant files that every story must clear, not story-level ATDDs.**

4. **"Verification-only" baseline stories are a repeatable pattern when the concern is threaded through feature stories.** Story 6.6 (accessibility) is the first validated instance: ARIA wired incrementally across 5.1/5.2/6.1; 6.6 was unskip + gap triage. **The same pattern applies to Epic 7 (error handling) and Epic 8 (governance) — if error-display UI and privacy invariants are threaded into earlier stories, Epic 7 and 8 become verification+gap-triage epics, not net-new-code epics.** Worth checking: is Epic 7's error copy already in place from Stories 3.7 / 6.5 (yes), or does 7.1/7.2/7.3 need to write new UI strings (no — `getSafeErrorMessage` + progress.js error branch already cover them)?

5. **Deferred-work is a platform-hardening backlog, not a per-story tail.** 20 items in Epic 6 alone, 40+ cumulative across Epics 3-6. They fall into three categories: (a) behavioural tests missing for live defensive code (NaN guards, re-entrancy, filename format); (b) cross-cutting infrastructure gaps (rate-limit, Cache-Control, axe-core, keyboard-nav); (c) cosmetic PR-body drift (surveyed, not actionable). Batch (a)+(b) into a dedicated platform-hardening story/mini-epic scheduled before Epic 7 ships (because Epic 7's error handling will generate more deferred items, and the backlog compounds). Category (c) is fully handled by memory rules — stop trying to patch Step 6 for cosmetic imprecision.

6. **Contract-drift-from-below is a real class of risk frontend work introduces.** Story 6.1 discovered the backend route was missing `generated_at`; the fix touched `src/routes/reports.js` AND amended `tests/epic4-4.3-get-api-reports-and-csv.atdd.test.js`. No malice, no mistake — but no gate, either. Epic 7 will touch `src/workers/**/*.js` error paths, potentially amending `getSafeErrorMessage` output contracts that `progress.js` reads. A "contract-amendment flag" in create-story and in PR-body would keep these visible. **Lightweight enough to ship as a memory rule + BAD Step 6 template tweak.**
