5. **Graceful-degradation email pattern in 3.6.** Worker marks `complete` BEFORE email; email failure leaves report intact. NFR-R3 satisfied by design, not by luck. Pattern worth reusing in future non-critical side-effects.

6. **Epic 2 retro commitments held.** Story spec commit gate: all 7 specs present in `_bmad-output/implementation-artifacts/`. MCP-Verified Endpoint Reference referenced at top of every story file. Multi-pass reviews continued — 3.1 had two review passes producing five patches.

## Challenges

1. **Distillate endpoint reference is not enough for non-trivial Mirakl behaviour.**
   Story 3.3 opened with "Endpoints verified against MCP-Verified Endpoint Reference (2026-04-18)" and still landed three wrong contracts. MCP was not connected in that specific session — a tool gap, not a BAD gap — but the lesson stands: a distillate pin catches field-name typos; it doesn't catch semantic errors like "which param accepts EANs vs SKUs" or "where channel context comes from". For anything where behaviour can't be statically verified, a live probe is the only reliable gate.

2. **PR body hallucination across 3.5 and 3.6.** BAD's Step 6 subagent fabricated files (`persistReport.js`, `generateCsv.js`), tables (`report_items`), and flags (`EMAIL_ENABLED`, retry/attachments logic) not present in the actual diff. 3.4 and 3.7 bodies were fine — so it's a partial pattern, not universal. The diff is the source of truth; the PR body has become decorative. No fix required at the BAD level as long as the reviewer audits diffs, but worth naming so nobody trusts future PR bodies unexamined.

3. **Sprint-status push race (same class as the epic-row tracking gap).** BAD's Step 7 pushes story-row `done` to main, then races the PR squash-merge and ends up stranded on local. PRs #42 and #43 required manual rebase-and-preserve to reconcile. At the epic level, `epic-3: in-progress` is still set in sprint-status.yaml despite all 7 stories being done. One root cause, two symptoms.

4. **Static-ordering ATDD checks produced a real code smell.** Stories 3.6 and 3.7 both ended up with `await import()` calls in the middle of `processJob` purely to satisfy "string X appears after string Y in source" scans. Runtime-correct, architecturally awkward. Tested twice now — it's a pattern. Static source-text scans are great for invariants (security, architecture) but create pressure to warp code when extended to ordering/sequencing.

5. **"DO NOT MODIFY" protected ATDD got modified twice.** Story 3.3 (active-filter check), Story 3.7 (async keyword on test callback). Both modifications were justified — the protected tests were wrong. The convention leaks when the protected contract is itself incorrect.

6. **Pre-emptive ATDD tests forced a manual `npm test` allowlist.** Phase 1 Epic-Start generated tests for 3.5/3.6/3.7 before their stories landed. Red tests blocked CI. Workaround: explicit file list in `package.json`. Drift-prone — every new pre-emptive test file requires an allowlist edit. Epic 4 will add 3 more.

7. **Epic 3 has never executed end-to-end with real traffic.** Unit and ATDD isolation only. Catalog-fetch → scan → score → persist → email has not been run against Gabriel's real catalog with real Redis + Resend. Epic 4.1 (POST /api/generate) is where the full pipeline activates for the first time — and where any Redis or Resend misconfiguration surfaces.

## Key Insights

1. **MCP distillate pin catches syntax; live probe catches semantics.** Epic 2 retro promoted MCP verification from guideline to hard gate. Epic 3 showed the gate is necessary but insufficient for semantic behaviour. For any new Mirakl endpoint (or new aspect of a known endpoint), live probe before the spec finalizes — not after review.

2. **BAD's autonomy curve is still climbing.** Test-quality self-correction (3.2 → 3.4+) and package.json test-scope maintenance both happened without instruction. The pipeline is learning — and the things it still misses (PR body accuracy, sprint-status push race) are mechanical gaps, not judgement gaps.

3. **Static source-text scans should stay close to invariants.** Security ("api_key never appears"), architecture ("no fetch outside apiClient.js") — perfect fit. Sequencing and ordering — creates code smells. A future ATDD design rule: if the check is "string X before string Y", either rewrite it as a runtime invariant or drop it.

4. **Process gaps reappear at different granularities.** "Done but untracked" showed up at Epic 2 level (Coolify deployment) and repeats at Epic 3 level (epic row) and story level (push race). Single root cause: no reconciliation step reads GitHub state back into sprint-status. Fix once, resolves three layers.

5. **Boring reliability compounds.** Two epics, zero security violations. Multi-pass code review continuing to catch real issues (3.1 transport-error retry, 3.7 SessionExpiredError sentinel). No heroics, no panic. The system is earning trust.
