## Challenges

1. **BAD Step 6 phrasing-drift now established across 4 consecutive PRs.** PR #60 ("Promise.all → allSettled" referring to unchanged Epic 3.3 code), PR #63 (story spec at `_bmad-output/planning-artifacts/stories/`, +80 lines vs actual +111, "27 test cases" vs actual 24, "5 additional test cases" vs actual 3+2), PR #64 ("40 test cases across 9 ACs" vs actual 36 across AC-1 through AC-7), PR #66 (no path-prefix hallucination — improvement). Memory rule `feedback_bad_pipeline_trust.md` continues to handle the risk at review time (diff is source of truth), but the deterministic guardrail (P5 from Epic 7 — diff-grep filename-claim check) shipped in prep PR #62 and is **not yet catching count-inflation claims** because P5 was scoped to file-path claims only. The phrasing pattern that recurs is **count overstatement** ("X tests", "Y AC labels", "+N lines"), not file-path drift. **Resolved in this retro — see Action Item P6.**

2. **Coordinator CWD-persistence bug surfaced through M1 gate false positive (Story 8.1 batch).** A `cd <worktree>` in an early Bash call persisted across the session; subsequent relative-path `sha256sum sprint-status.yaml` reads sampled the WORKTREE copy (always `backlog`, never touched by Phase 2 steps), not the repo-root copy. The hash-mismatch fired M1 alarm; manual investigation revealed apples-to-oranges sampling. Spurious `git commit` on story-8.2 branch (commit `c492374`) cleaned up via `git reset --hard HEAD~1`, never pushed to origin. Cost: ~20 min investigation + one cleanup. **This is a class-of-bug, not an isolated incident** — any Phase 2 step using relative paths after a `cd` is at risk. The symmetrical rule already exists in `bad-review/references/merge-procedure.md`. The BAD coordinator side has no equivalent guard. **Resolved in this retro — see Action Item P7.**

3. **P2 gate (clean-tree post-Step-3) — taxonomy refinement needed.** Gate fired on Story 8.1 with ` M _bmad-output/implementation-artifacts/8-1-hourly-ttl-deletion-cron.md`. Investigation showed working tree was BEHIND HEAD (spec file reverted from `Status: review` post-dev to `Status: ready-for-dev` pre-dev), not AHEAD. The original Epic 7 incident the gate was designed for was working-tree-AHEAD-of-HEAD (uncommitted dev-agent output). Both fail the `git status --porcelain` check identically, but the semantic is opposite: AHEAD = real defect (lost work), BEHIND = silent regression of an already-committed file by some downstream action. Two distinct failure modes now visible. Recovery was mechanical (`git checkout <file>`); no real defect. **Resolved in this retro — see Action Item P8.**

4. **Step 2 (ATDD subagent) wrote sprint-status.yaml to the WORKTREE copy and committed to the branch.** Story 8.1 Step 2 commit `82e7ded` ("scaffold sufficient — no new tests; set atdd-done in sprint-status") staged `_bmad-output/implementation-artifacts/sprint-status.yaml` from the worktree path with the stale `atdd-done` value. Step 7 `/code-review:code-review` caught this when diffing PR branch against main — PR #63 HEAD `152c156` was corrected before merge. Had it slipped, merging would have overwritten main's correct `review`/`done` state with the stale branch value, silently regressing the sprint-status ledger. Skill rule clearly states "Update sprint-status.yaml at the REPO ROOT (not the worktree copy)" but the Step 2 subagent interpreted "update sprint-status.yaml" as a single relative-path operation. **No deterministic guardrail prevents this today.** Same root cause as Challenge #2 (relative-path inside worktree). **Resolved in this retro — see Action Item P9.**

5. **Static-asset (`@fastify/static`) bypass of rate-limit deferred.** Step 7 review of PR #66 surfaced that `staticPlugin` registers BEFORE `rateLimit` in `src/server.js`, so static asset fetches (`/css/*`, `/js/*`, `/`, etc.) bypass the 60 req/min global default. Pre-existing in the registration order; out of scope for Story 8.3 per the spec's "route-layer only" framing. Acceptable for MVP because static assets are tiny (~few KB each), CDN-friendly, and already cacheable at the browser level — abuse vector is bandwidth-only, not DoS-amplification. Annotated in deferred-work.md (PR #66 entry). **Disposition: defer post-MVP.** Not a blocker for ship; revisit if static-asset abuse becomes observable.

6. **PR body count-inflation continued in 8.x family.** Epic 8 Challenge — same class as Epic 7 Challenge #7 — three PRs (#63, #64, #66) carried at-least-one count overstatement. P5 deterministic file-path check shipped in #62 caught zero of these because they were not file-path claims. The pattern is structurally consistent: when a Step 6 prompt summarises in connected prose, comma-separated and conjunction-glued claims get rounded for narrative flow. **Concrete retro action-item: "one claim per sentence" constraint in Step 6 prompt.** Forces the LLM to commit each claim as a discrete proposition, which flattens the rounding tendency. **See Action Item P6.**

7. **AC-5 spec ambiguity surfaced in PR #64 review (deferred, not blocking).** Story 8.2 AC-5 reads "job_id never in report URL or response" — the inverse (report_id never in jobs response) is intentionally NOT asserted, because `getJobStatus()` returns `{ status, phase_message, report_id }` by design (Story 4.2a requires it for the frontend redirect). A future reader could "fix" the perceived asymmetry by stripping `report_id` from the jobs response and breaking the redirect flow. Deferred-work.md captures this. **Resolution belongs to a post-MVP epics.md pass to amend AC-5 wording, not Epic 8.** No code-side action needed.

## Key Insights

1. **Named prep PR is now empirically the load-bearing artifact of the compounding-retro loop.** Epics 3→4, 4→5, 5→6 compounded with prep PRs; Epic 6→7 broke without one; Epic 7→8 restored it via the M2 codification + the actual shipping of PR #62. This is now a 4-of-5 pattern with one explicit counter-example. **The retro itself is not the deliverable** — the named prep PR carrying the retro's action items, merged before the next epic's first story, is. Codify this further: any future retro that produces process action items must end with a "prep PR scope" section that lists items, owners, deadline (= "before next epic's Story 1 create-story"), and a single named PR title. The retro is incomplete without it.

2. **Two new failure modes attributable to relative-path-inside-worktree.** Challenge #2 (coordinator CWD-persistence — false M1 alarm + branch pollution) and Challenge #4 (Step 2 worktree-copy sprint-status write — almost-regressed main's ledger) have a common root: relative paths interpreted against whatever CWD or worktree the actor happened to land in. The merge-procedure side learned this lesson; the BAD coordinator side and the BAD step subagents (especially Step 2) have not. **A single coordinator-wide rule + per-step prompt insertion ("absolute paths or `git -C <absolute>` for all sprint-status reads/writes") would close both classes in one move.** Cheap, mechanical, no false-positive risk.

3. **Step 6 phrasing drift is structurally count-inflation, not file-path drift, and is structurally connected-prose-driven.** Three of the four PR #63/#64/#66 hallucinations were count overstatements bundled into compound sentences ("This adds X tests across Y ACs covering Z scenarios"). The "one claim per sentence" constraint forces the LLM to commit each claim as a discrete proposition the audit-trail can then verify against the diff. P6 below codifies this. Less ambitious than a deterministic regex (because count claims are harder to extract reliably than path-like strings), but addresses the actual pattern that has now persisted across 4 PRs despite memory rules.

4. **The verification-only baseline epic pattern is now reliable across three epics.** Epic 6.6 (one story), Epic 7 (three stories), Epic 8 (three stories — 8.2 is pure verification, 8.3 is route-layer-thin verification, 8.1 is one new module). For governance/hardening/edge-case epics, default to scaffold-with-deliberate-failures + thin-implementation. Story sizes stay small, refactor risk stays near zero, ATDD scaffolds double as the implementation spec when the epic is verification-heavy. **For any post-MVP work that resembles platform hardening or governance, plan to this pattern.**

5. **Sprint-change-proposal as a mid-epic story-addition mechanism works cleanly.** Story 8.3 was added mid-Epic-8 via [sprint-change-proposal-2026-04-23.md](../planning-artifacts/sprint-change-proposal-2026-04-23.md). The proposal forced explicit MVP-vs-post-MVP triage (4-in / 5-out) and provided a paper trail for the three-artifact edit (epics.md + epics-distillate.md + sprint-status.yaml). **This is the right mechanism for promoting backlog items into the active epic** — better than amending an existing story (state-machine violation) and better than trickling into "we'll do it later" (Epic 6 T1 failure mode). Adopt for any post-MVP retro that produces in-scope-promotion findings.

6. **Epic 8 ships the MVP. Deferred-work backlog is bounded and triaged.** ~32 items entered Epic 8; ~10 closed/scoped-out by 8.3; ~12 added by Epic 8 review; ~34 carry forward to post-MVP. Of the carry-forward, the majority are either (a) belt-and-suspenders behavioural assertions for already-passing static-scan invariants, (b) structural-fragility nits (literal-string-match assertions, header duplication), or (c) explicitly post-MVP UX/accessibility work. **No carry-forward item is a known production defect or a gating risk for ship.** The MVP is ship-ready functionally; the pre-ship checklist is operational verification only (see Pre-MVP Ship Checklist below).

## Action Items

### Process Improvements

**P6. "One claim per sentence" constraint at BAD Step 6 prompt.** Owner: Pedro. Add to the Step 6 (PR body construction) prompt — both in the BAD coordinator and in `bad-review`'s Phase 1 PR-body audit step — an explicit instruction: "Each factual claim about the diff (file count, line count, test count, AC count, behaviour change) must be expressed as a single declarative sentence with one verb and one count or one file. Do not bundle multiple counts into one sentence ('adds X tests across Y ACs covering Z scenarios' is forbidden). Each claim must be independently verifiable against the diff." Rationale: 4-PR pattern of count-inflation; structural connected-prose driver; deterministic regex extraction is harder than path-claim extraction so a constitutional prompt rule is the cheaper move. Ship as a standalone memory + skill prompt edit; no code change required.

**P7. Coordinator absolute-path-or-`git -C` discipline for all sprint-status reads/writes.** Owner: Pedro. Add a coordinator-level rule (in `.claude/skills/bad/SKILL.md` or a referenced common-rules file) requiring: ALL `sha256sum`, `git status`, `git add`, `git commit`, `cat`, `grep`, or `Read`/`Edit` operations on `_bmad-output/implementation-artifacts/sprint-status.yaml` MUST use the absolute path `D:/Plannae Project/DynamicPriceIdea/_bmad-output/implementation-artifacts/sprint-status.yaml` OR be invoked via `git -C D:/Plannae\ Project/DynamicPriceIdea ...`. Never relative. Apply identically to dependency-graph.md and any other repo-root state file. Rationale: Challenge #2 (coordinator CWD bug) and Challenge #4 (Step 2 worktree-copy write) share this root cause. The merge-procedure side already learned it; close the symmetry. **Single rule, two failure-mode closures.**

**P8. P2 gate taxonomy refinement — distinguish working-tree-AHEAD-of-HEAD from working-tree-BEHIND-HEAD.** Owner: Pedro. Extend the Step 3/4 clean-tree gate logic: after detecting non-empty `git status --porcelain`, run `git diff HEAD --stat` and inspect:
- If diff shows new content (additions, untracked files) → working tree is AHEAD → original failure mode → HALT (current behaviour).
- If diff shows reverted content (working tree is a subset/earlier version of HEAD) → working tree is BEHIND → different class → auto-recover via `git checkout <files>` and log as a taxonomy-refinement observation; do NOT HALT.

Rationale: Challenge #3 — both modes fail `git status --porcelain` identically but mean opposite things. Auto-recovering BEHIND case avoids the spurious HALT cost without reducing AHEAD coverage. Update `feedback_bad_step3_clean_tree_gate.md` to reflect the two-mode taxonomy.

**P9. Step 2 (ATDD subagent) prompt: hard-prohibit `git add` of sprint-status.yaml in worktree.** Owner: Pedro. Add to the Step 2 prompt: "You MUST NOT `git add` `_bmad-output/implementation-artifacts/sprint-status.yaml` to the worktree branch. Sprint-status updates always target the REPO ROOT path absolute, and are committed to the repo-root main branch via Step 7 reconciliation. If your tooling stages it implicitly (e.g. `git add .`), you must `git restore --staged sprint-status.yaml` before commit." Rationale: Challenge #4. The current "Update sprint-status.yaml at REPO ROOT (not the worktree copy)" wording is ambiguous if the agent interprets "update" as a single relative-path operation. Hard-prohibition language plus the explicit unstage instruction is unambiguous.
